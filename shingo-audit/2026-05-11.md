## Scope summary

The shingo program is a Solana / Anchor program (1,362 lines) plus two Arcium MPC circuit definitions (62 lines) that monetizes paid access to encrypted trading signals. Traders run seasons of encrypted signals, subscribers pay sol into a per-season escrow PDA, decrypt episodes during the active season via Arcium, and reveal them publicly after close. The audit target is HEAD commit `2a204b0`, which introduced escrow, spots, and minimum-episodes logic.

### Files in scope

| File | Lines | Role |
|---|---:|---|

| `encrypted-ixs/src/lib.rs` | 62 | Arcium MPC circuits `decrypt_signal` and `reveal_signal` |

The build artifacts at `build/*.arcis` are out of scope as source review. Their on-chain `circuit_hash!()` is in scope as an integrity-verification mechanism. The source-URL pinning is in scope and discussed under M-06.

Dependencies (`Cargo.toml`):
- `anchor-lang` 0.32.1 with `init-if-needed` feature
- `arcium-{client,macros,anchor}` 0.9.7
- `solana-security-txt` 1.1.2
- `bytemuck` 1.24.0

Build profile is hardened: `overflow-checks = true`, `lto = "fat"`, `codegen-units = 1`. Silent integer wrapping is therefore unlikely on release, but the program also uses `checked_add` and `checked_div` defensively, so arithmetic bugs surface as panics or explicit error variants.

### Methodology

The audit was conducted as a manual review on the full 1,424 lines. All findings were verified by reading the source. Code excerpts quote the file at HEAD commit `2a204b0`. Recommended fixes are sketches, not normative patches. M-08 (computation_offset DoS) was additionally verified against `arcium-client 0.9.7`'s `pda.rs` to confirm the comp PDA derivation does not namespace by caller.

### Trust model

- `DEVELOPER_ADDRESS` (`HhEBDdSK7ywsesAFdMcsQjWiWVBTYbjS386TJAVibMJQ`) is a hardcoded constant pubkey. It receives the 1% per-subscription tip and an auto-issued `shingo_pass` for every season (granting it full decrypt rights). Trusted.
- The Arcium Cerberus MPC cluster is trusted to honestly execute the encrypted circuits. The program verifies signed callback outputs via `output.verify_output(cluster_account, computation_account)` in both decrypt and reveal callbacks. The `cluster_account` itself is address-constrained to `derive_cluster_pda!(mxe_account, …)`. This trust path is correctly implemented at the framework boundary.
- Traders are trusted to encrypt real signals. There is no on-chain validation of ciphertext contents.
- Subscribers are not trusted. The protocol's privacy and payment model depends on `decrypt_signal` and `reveal_signal` correctly binding the subscriber's pass to the signal they request. This binding is broken (findings C-07 and C-08).

## Critical findings

### C-01 `season.trader` is never assigned, `close_season` is permanently unreachable

**Severity**: Critical.
**Files**: `programs/shingo/src/lib.rs:204` (declaration), `:889-895` (init), `:982` (read).

The `Season` struct declares a `trader: Pubkey` field as its first member:

```rust
#[account]
#[derive(InitSpace)]
pub struct Season {
    pub trader: Pubkey,
    pub subscription_price: u64,
    pub id: u64,
    pub is_active: bool,
    pub count: u64,
    pub minimum_number_of_episodes: u64,
    pub spots: u64,
    pub subscribers: u64,
}
```

The `initialize_season` function (L860-915) sets every other field but never assigns `season.trader`. A whole-program grep confirms zero write sites for `season.trader`. Anchor's `init` macro zeroes the account data, so for every season the field is `Pubkey::default()` (the 32-byte zero address).

The `close_season` function (L981-984) reads this field as its access-control gate:

```rust
require!(
    ctx.accounts.season.trader.eq(ctx.accounts.trader.key),
    ShingoProgramError::Nono
);
```

The comparison is `Pubkey::default().eq(real_signer_pubkey)`. For any real signer (any user with a private key) the comparison returns false and the `require!` fires `ShingoProgramError::Nono`. No one can close a season.

#### Impact

Every season opened on the program is in a one-way state. Subscribers can pay into the escrow (once C-05 is fixed) but the trader cannot withdraw. Funds are permanently locked. The reveal flow remains unreachable too, since `reveal_signal` requires `!season.is_active` and `is_active` can only flip in `close_season`.

#### Proof of concept

Pseudo-test (Anchor, devnet):

```typescript
await program.methods.initializeTraderAccount().rpc();
await program.methods.initializeSeason(new BN(1_000_000), new BN(100), new BN(1)).rpc();
await program.methods.encryptSignal(...).rpc(); // count >= minimum
await expect(program.methods.closeSeason().rpc())
    .to.be.rejectedWith(/Nono/);
```

#### Recommended fix

In `initialize_season`, set the trader field at init time alongside the other assignments:

```rust
let season = &mut ctx.accounts.season;
season.trader = ctx.accounts.trader.key();   // add this
season.subscription_price = subscription_price;
...
```

Better: enforce the binding at the constraint level by adding `has_one = trader` on the `Season` declarations in `CloseSeason`, `EncryptSignal`, and `RevealSignal` so the framework catches mismatches without a separate body check.

Note that fixing C-01 alone does not make `close_season` work. Findings C-02 and C-03 remain.

### C-02 `close_season` CPI signer seeds omit `season_id`, drain transfer cannot match the escrow PDA

**Severity**: Critical.
**Files**: `programs/shingo/src/lib.rs:1003-1012` (signer seeds), `:364-370` (init seeds), `:464-468` (re-derivation seeds).

The `SeasonEscrow` PDA is initialized in `InitializeSeason` with three seed components plus bump:

```rust
seeds = [SeasonEscrow::SEED, trader.key().as_ref(),
    trader_account.current_season.checked_add(1).unwrap_or(trader_account.current_season).to_le_bytes().as_ref()]
```

The `CloseSeason` account context re-derives the PDA with three components too:

```rust
seeds = [SeasonEscrow::SEED, trader.key().as_ref(),
    trader_account.current_season.to_le_bytes().as_ref()]
```

Inside the function body the signer seeds are constructed with **two** components plus bump (the season number is missing):

```rust
let seeds = &[
    SeasonEscrow::SEED,
    transaction_signer_key.as_ref(),
    &[ctx.accounts.season_escrow.bump],
];
let signer_seeds = &[&seeds[..]];
```

When `system_program::transfer` is invoked with `CpiContext::new_with_signer(…, signer_seeds)`, the Solana runtime calls `Pubkey::create_program_address(signer_seeds, program_id)` to verify that the resulting address matches the `from` account (`season_escrow.key()`). It does not. The CPI fails with an unauthorized-signer error.

#### Impact

Independent of C-01, the escrow drain CPI cannot succeed. Even if `season.trader` were assigned correctly, this CPI rejects.

#### Recommended fix

Include the missing seed component:

```rust
let season_number_le = season.id.to_le_bytes();   // or trader_account.current_season
let seeds = &[
    SeasonEscrow::SEED,
    transaction_signer_key.as_ref(),
    season_number_le.as_ref(),
    &[ctx.accounts.season_escrow.bump],
];
```

A cleaner pattern is to use the Anchor `close = trader` directive on the escrow account in the `CloseSeason` context, which lets Anchor handle the lamport movement and rent reclamation without a manual CPI. See the joint fix in C-03.

### C-03 `close_season` uses `system_program::transfer` on a program-owned account, transfer is rejected

**Severity**: Critical.
**Files**: `programs/shingo/src/lib.rs:1014-1026` (transfer), `:364-371` (init).

The escrow account is initialized with Anchor's `init`:

```rust
#[account(
    init,
    payer = trader,
    space = 8 + SeasonEscrow::INIT_SPACE,
    seeds = [SeasonEscrow::SEED, trader.key().as_ref(), …],
    bump
)]
pub season_escrow: Account<'info, SeasonEscrow>,
```

Anchor's `init` calls `system_program::create_account` and assigns the new account's owner to the calling program. Post-init, `season_escrow.owner == shingo_program_id`, not `system_program::ID`. The account also carries 8 bytes of discriminator plus 1 byte for the `bump` field.

The drain in `close_season` (L1016-1026) is:

```rust
let all_the_money_from_the_season = ctx.accounts.season_escrow.get_lamports();

system_program::transfer(
    CpiContext::new_with_signer(
        ctx.accounts.system_program.to_account_info(),
        system_program::Transfer {
            from: ctx.accounts.season_escrow.to_account_info(),
            to: ctx.accounts.trader.to_account_info(),
        },
        signer_seeds,
    ),
    all_the_money_from_the_season,
)?;
```

Solana's System program enforces two preconditions on the `from` account of a `Transfer` instruction:

1. `from.owner == system_program::ID`. The escrow is program-owned, so this fails.
2. `from.data_len() == 0`. The escrow carries `8 + 1 = 9` bytes of data, so this fails too.

Either condition is sufficient to reject the transfer. The correct pattern for moving lamports out of a program-owned PDA is direct lamport mutation, which does not need a CPI:

```rust
**ctx.accounts.season_escrow.to_account_info().try_borrow_mut_lamports()? -= amount;
**ctx.accounts.trader.to_account_info().try_borrow_mut_lamports()? += amount;
```

Alternatively, declare `#[account(mut, …, close = trader)]` on the escrow in the `CloseSeason` context and let Anchor handle the lamport movement and rent reclamation atomically.

#### Impact

Independent of C-01 and C-02, the system-program rejects the transfer. The combination of C-01, C-02, and C-03 means that fixing any one of them leaves the function broken in two other ways. A single coherent rewrite of the close path is required.

#### Recommended fix

Replace the manual transfer with the Anchor close directive plus a rent-aware structure:

```rust
#[derive(Accounts)]
pub struct CloseSeason<'info> {
    pub system_program: Program<'info, System>,
    #[account(mut)]
    pub trader: Signer<'info>,
    #[account(
        mut,
        seeds = [TraderAccount::SEED, trader.key().as_ref()],
        bump,
        has_one = trader  // future-proof if you add a trader field to TraderAccount
    )]
    pub trader_account: Account<'info, TraderAccount>,
    #[account(
        mut,
        seeds = [Season::SEED, trader.key().as_ref(), trader_account.current_season.to_le_bytes().as_ref()],
        bump,
        has_one = trader,                       // requires C-01 fix
        close = trader,                          // reclaim Season rent (optional, can keep open for history)
    )]
    pub season: Account<'info, Season>,
    #[account(
        mut,
        seeds = [SeasonEscrow::SEED, trader.key().as_ref(), trader_account.current_season.to_le_bytes().as_ref()],
        bump,
        close = trader,                          // drain escrow + reclaim rent in one move
    )]
    pub season_escrow: Account<'info, SeasonEscrow>,
}
```

With `close = trader` on `season_escrow`, the entire balance moves to `trader` and the account is deallocated. Remove the manual `system_program::transfer` from the function body.

If keeping `season` open for historical reasons (since signals reference it), drop the `close = trader` on `season` and only close `season_escrow`.

### C-04 `close_season` minimum-episodes guard is inverted

**Severity**: Critical (post-fix), High (latent today).
**File**: `programs/shingo/src/lib.rs:989-992`.

```rust
require!(
    ctx.accounts.season.count < ctx.accounts.season.minimum_number_of_episodes,
    ShingoProgramError::CannotCloseSeasonUntilMinimumNumberOfEpisodesIsReached
);
```

The error variant says "Cannot close a season until it reaches its minimum number of episodes". The intended semantic is "close is allowed when `count >= minimum`". The coded predicate is the opposite: close is allowed **only while** `count < minimum`.

#### Impact

Once C-01, C-02, and C-03 are fixed, a trader can:

1. Open a season with `minimum_number_of_episodes = 1000`.
2. Attract subscribers (each pays `subscription_price` into escrow).
3. Publish 0 to 999 signals.
4. Call `close_season`. The guard passes because `count < minimum` (e.g., `0 < 1000`).
5. Drain the escrow.

The trader pays subscribers nothing in exchange. Conversely, an honest trader who delivers `count >= minimum` is permanently blocked from closing their season, locking subscriber funds.

Today the bug is masked because C-01 and C-02 and C-03 prevent any close from succeeding. The moment those are patched in the natural direction, this rug primitive activates.

#### Proof of concept

```typescript
// Setup: trader opens season with minimum=10, no signals published.
await traderProgram.methods
    .initializeSeason(new BN(1_000_000_000), new BN(1000), new BN(10))
    .rpc();
// Subscriber pays.
await subscriberProgram.methods.subscribeToSeason()
    .accounts({ trader: trader.publicKey, ... })
    .rpc();
// Trader closes without publishing any signal. Should fail, but with C-04
// inverted and C-01, C-02, C-03 patched, it succeeds.
await traderProgram.methods.closeSeason().rpc();
// Trader now holds the subscriber's deposit.
```

#### Recommended fix

```rust
require!(
    ctx.accounts.season.count >= ctx.accounts.season.minimum_number_of_episodes,
    ShingoProgramError::CannotCloseSeasonUntilMinimumNumberOfEpisodesIsReached
);
```

Pair this with H-06 (require `minimum_count > 0` in `initialize_season`) to prevent traders setting `minimum=0` and rugging immediately.

### C-05 `subscribe_to_season` spot-cap guard is inverted

**Severity**: Critical.
**File**: `programs/shingo/src/lib.rs:924-927`.

```rust
require!(
    ctx.accounts.season.spots < ctx.accounts.season.subscribers,
    ShingoProgramError::SeasonMaximumSubscribersNumberReached
);
```

At season open, `spots = N > 0` and `subscribers = 0` (the latter never changes, see H-01). The predicate `N < 0` is always false. The `require!` always fires `SeasonMaximumSubscribersNumberReached`.

#### Impact

No user can ever subscribe to any season. The protocol's entire revenue path is dead. Combined with C-01 through C-03, the program is effectively inert: no value enters or leaves.

#### Proof of concept

```typescript
await program.methods.initializeTraderAccount().rpc();
await program.methods.initializeSeason(new BN(1_000), new BN(100), new BN(1)).rpc();

const subscribeIx = await subscriberProgram.methods.subscribeToSeason()
    .accounts({ trader: trader.publicKey, ... })
    .instruction();

await expect(sendAndConfirm(subscribeIx))
    .to.be.rejectedWith(/SeasonMaximumSubscribersNumberReached/);
```

#### Recommended fix

```rust
require!(
    ctx.accounts.season.subscribers < ctx.accounts.season.spots,
    ShingoProgramError::SeasonMaximumSubscribersNumberReached
);
```

Pair with H-01: increment `season.subscribers` in `subscribe_to_season` so the cap actually binds.

### C-06 `EncryptSignal.season` is unconstrained, any signer corrupts any trader's season count

**Severity**: Critical.
**File**: `programs/shingo/src/lib.rs:474-491`.

```rust
#[derive(Accounts)]
pub struct EncryptSignal<'info> {
    pub system_program: Program<'info, System>,

    #[account(mut)]
    pub trader: Signer<'info>,

    #[account(mut)]
    pub season: Account<'info, Season>,    // no seeds, no has_one

    #[account(
        init_if_needed,
        payer = trader,
        space = 8 + Signal::INIT_SPACE,
        seeds = [Signal::SEED, trader.key().as_ref(), season.id.to_le_bytes().as_ref(), season.count.to_le_bytes().as_ref()],
        bump
    )]
    pub signal: Account<'info, Signal>,
}
```

`season` is mutable but has no seeds, no `has_one`, and no body-level check that it belongs to the signer. The function body increments `season.count`:

```rust
let season = &mut ctx.accounts.season;
let new_count = season.count.checked_add(1).ok_or(...)?;
season.count = new_count;
```

#### Impact

Any signer Mallory can call `encrypt_signal` passing victim trader Alice's `Season` account. Mallory's own signature is required, but the season account is not bound to Mallory. The function:

1. Creates a `Signal` account at PDA `[Signal::SEED, Mallory.key(), Alice.season.id, Alice.season.count]`. This is Mallory's address space, paid out of Mallory's wallet.
2. Increments Alice's `season.count`.

The harm is in (2). Alice's signal index space now has a gap at the count Mallory observed before the increment. Alice's next `encrypt_signal` call writes a Signal at the bumped count, and any off-chain indexer iterating slots `0..season.count` will find the slot Mallory bumped missing.

Compound interactions:

- Combined with C-04 once the direction is corrected (`count >= minimum`), Mallory can push Alice's `season.count` past her `minimum_number_of_episodes`, allowing Alice to close prematurely. Alice still has to sign `close_season` herself, so Mallory cannot trigger the rug, but they can enable it.
- Combined with M-02 (`init_if_needed` on the signal account), if a future code path ever decrements `season.count`, Mallory's signal at the older slot could be silently overwritten by Alice's legitimate signal.

#### Proof of concept

```typescript
// Alice opens a season, no signals yet.
await aliceProgram.methods.initializeSeason(new BN(1000), new BN(10), new BN(5)).rpc();
const beforeCount = (await program.account.season.fetch(aliceSeason)).count;
// expect: 0

// Mallory calls encrypt_signal with Alice's season.
await malloryProgram.methods.encryptSignal(...)
    .accounts({ trader: mallory.publicKey, season: aliceSeason, ... })
    .rpc();

const afterCount = (await program.account.season.fetch(aliceSeason)).count;
// observed: 1
expect(afterCount.toNumber()).to.equal(1);
```

#### Recommended fix

Bind `season` to the signer:

```rust
#[account(
    mut,
    seeds = [Season::SEED, trader.key().as_ref(), season.id.to_le_bytes().as_ref()],
    bump,
    has_one = trader,                      // requires C-01 fix
    constraint = season.is_active @ ShingoProgramError::SeasonIsNotActive,  // see H-04
)]
pub season: Account<'info, Season>,
```

### C-07 `RevealSignal.season` is not bound to signal author, any user can reveal another trader's private signal

**Severity**: Critical (post C-01 fix), High (latent today).
**Files**: `programs/shingo/src/lib.rs:691-694, 1248-1258`.

```rust
#[derive(Accounts)]
pub struct RevealSignal<'info> {
    pub season: Box<Account<'info, Season>>,             // no seeds, no has_one
    pub signal: Box<Account<'info, Signal>>,             // no seeds, no has_one
    ...
}

pub fn reveal_signal(ctx: …, …) -> Result<()> {
    require!(
        (ctx.accounts.season.id == ctx.accounts.signal.metadata.season_id)
            && !ctx.accounts.season.is_active,
        ShingoProgramError::SignalCannotBeRevealedBecauseItsSeasonNotFinished
    );
    ...
}
```

The require checks two things: that the season's `id` matches the signal's stored `metadata.season_id`, and that the season is inactive. `season_id` is a per-trader auto-incrementing counter starting at 1. Two different traders both run their respective "season 1", "season 2", and so on. The require accepts any closed `Season` account whose `id` happens to match the signal's `season_id`, regardless of which trader the season actually belongs to.

#### Impact

Once C-01, C-02, and C-03 are fixed so close_season works, attacker Mallory can:

1. Open her own trader account and run `initialize_season` once to create her season 1.
2. Close her season 1 (legal once she meets minimum; with H-06 unfixed she can pick `minimum_count = 0` and close immediately after delivering zero signals).
3. Call `reveal_signal` passing her closed season 1 plus victim Alice's still-encrypted signal whose `metadata.season_id == 1`.
4. The require passes (`1 == 1`, `!is_active`). Arcium decrypts Alice's signal. The plaintext is written to `RevealedSignal` (keyed at Alice's PDA) and emitted as a `ClearSignal` event.

The privacy guarantee that subscribers paid for is broken by anyone who can spin up a same-id closed season. Even if Alice has paying subscribers expecting early access, the moment any same-id closed season exists, anyone can leak Alice's plaintexts publicly.

#### Recommended fix

Bind `season` to the signal's author via the PDA derivation:

```rust
#[account(
    seeds = [Season::SEED, signal.metadata.author.as_ref(), signal.metadata.season_id.to_le_bytes().as_ref()],
    bump,
    constraint = !season.is_active @ ShingoProgramError::SignalCannotBeRevealedBecauseItsSeasonNotFinished,
)]
pub season: Box<Account<'info, Season>>,
```

This ensures that only the actual author's closed season for that season_id can be passed. Cross-trader substitution is impossible.

### C-08 `DecryptSignal.signal` and `trader_account` are unconstrained, one subscription decrypts every signal

**Severity**: Critical (post C-05 fix), High (latent today).
**Files**: `programs/shingo/src/lib.rs:545-561, 1113-1166`.

The `DecryptSignal` context lacks binding between the `signal` account, the `season` account, the `trader_account`, and the `follower_pass`:

```rust
#[queue_computation_accounts("decrypt_signal", follower)]
#[derive(Accounts)]
#[instruction(computation_offset: u64)]
pub struct DecryptSignal<'info> {
    pub season: Box<Account<'info, Season>>,                // unconstrained
    #[account(mut)]
    pub follower: Signer<'info>,
    pub trader: SystemAccount<'info>,                       // unconstrained
    pub trader_account: Box<Account<'info, TraderAccount>>, // unconstrained
    #[account(seeds = [SubscriptionPass::SEED, follower.key().as_ref(), trader.key().as_ref(), trader_account.current_season.to_le_bytes().as_ref()], bump)]
    pub follower_pass: Box<Account<'info, SubscriptionPass>>,
    pub signal: Box<Account<'info, Signal>>,                // unconstrained
    ...
}
```

The function body checks only `follower_pass.owner == follower.key`. The MPC then decrypts whatever `signal` account was passed, regardless of whether it belongs to the trader/season the follower paid for.

#### Impact

Attacker Mallory needs only one valid subscription anywhere. Once C-05 is fixed and subscriptions work, Mallory can:

1. Subscribe to the cheapest available season (cost: `subscription_price + tip` lamports, can be as low as a few thousand lamports).
2. For every signal she wants to read, call `decrypt_signal` passing:
   - `follower_pass`: her legitimate pass for her cheap trader's current season.
   - `trader`, `trader_account`: the cheap trader and their TA.
   - `signal`: the target trader's premium signal she wants to read.
3. The Arcium MPC decrypts the target signal, re-encrypted for Mallory's chosen `receiver` pubkey. The `ObservableSignal` event delivers the plaintext to Mallory.

The protocol's entire revenue and content-access model is bypassed by a single cheap subscription. Every premium signal across every trader is freely readable.

#### Recommended fix

Bind every relevant account to a single source of truth, the `trader.key()`:

```rust
#[derive(Accounts)]
#[instruction(computation_offset: u64)]
pub struct DecryptSignal<'info> {
    pub trader: SystemAccount<'info>,

    #[account(
        seeds = [TraderAccount::SEED, trader.key().as_ref()],
        bump,
    )]
    pub trader_account: Box<Account<'info, TraderAccount>>,

    #[account(
        seeds = [Season::SEED, trader.key().as_ref(), signal.metadata.season_id.to_le_bytes().as_ref()],
        bump,
        has_one = trader,                                            // requires C-01 fix
    )]
    pub season: Box<Account<'info, Season>>,

    #[account(
        seeds = [Signal::SEED, trader.key().as_ref(), signal.metadata.season_id.to_le_bytes().as_ref(), signal.metadata.number.to_le_bytes().as_ref()],
        bump,
        constraint = signal.metadata.author == trader.key() @ ShingoProgramError::NotSubbed,
    )]
    pub signal: Box<Account<'info, Signal>>,

    #[account(
        seeds = [SubscriptionPass::SEED, follower.key().as_ref(), trader.key().as_ref(), signal.metadata.season_id.to_le_bytes().as_ref()],
        bump,
    )]
    pub follower_pass: Box<Account<'info, SubscriptionPass>>,

    ...
}
```

Note the keying change on `follower_pass`: derive from `signal.metadata.season_id` instead of `trader_account.current_season`. This both binds the pass to the requested signal's season and fixes H-07 (subscribers locked out after season transition).

### C-09 `initialize_trader_account` resets state on `init_if_needed`, trader bricks subscriber funds

**Severity**: Critical (subscriber fund-loss).
**Files**: `programs/shingo/src/lib.rs:346-353, 838-850`.

```rust
#[account(
    init_if_needed,
    payer = trader,
    space = 8 + TraderAccount::INIT_SPACE,
    seeds = [TraderAccount::SEED, trader.key().as_ref()],
    bump
)]
pub trader_account: Account<'info, TraderAccount>,

...

pub fn initialize_trader_account(ctx: Context<InitializeTraderAccount>) -> Result<()> {
    let trader_account = &mut ctx.accounts.trader_account;
    trader_account.current_season = 0;
    trader_account.has_active_season = false;
    trader_account.signal_count = 0;
    emit!(NewTrader { public_key: ctx.accounts.trader.key() });
    Ok(())
}
```

`init_if_needed` succeeds silently when the account already exists, and the function unconditionally writes the three reset assignments.

#### Impact

Trader Alice, after running a season for a while:

- `Alice.trader_account.current_season = 3`
- `Alice.trader_account.has_active_season = true`
- Alice's season 3 has paying subscribers, escrow holds N SOL.

Alice (or, via UI bug, anyone who can craft this transaction in her name) calls `initialize_trader_account` again. State after the call:

- `Alice.trader_account.current_season = 0`
- `Alice.trader_account.has_active_season = false`

The on-chain Season 3 PDA and SeasonEscrow 3 PDA still exist with the subscribers' funds. But `close_season` derives both account seeds from `trader_account.current_season`, which is now 0. The PDA `[Season::SEED, Alice.key(), 0]` does not exist, so Anchor deserialization fails. There is no path back to closing season 3 and recovering the escrow.

Subscribers lose N SOL. Alice gains nothing (she paid only the re-init rent cost). This is a denial-of-service against subscribers carried out by the trader, intentionally or accidentally.

Note: even if Alice can recover by calling `initialize_season` again to push `current_season` back to 3, that fails too. `initialize_season` derives `[Season::SEED, trader, current_season+1]` and tries to `init` it. After the reset, `current_season=0`, so it tries `[Season::SEED, Alice, 1]`. If Alice's original season 1 still exists at that PDA (a different past season), Anchor rejects with "account already in use".

#### Recommended fix

Change `init_if_needed` to plain `init`. The instruction's doc comment already says it is called once per trader, so re-initialization is not intended.

```rust
#[account(
    init,
    payer = trader,
    space = 8 + TraderAccount::INIT_SPACE,
    seeds = [TraderAccount::SEED, trader.key().as_ref()],
    bump
)]
pub trader_account: Account<'info, TraderAccount>,
```

### M-05 No `close = recipient` directives, rent permanently locked

**Severity**: Medium (operational).
**File**: every `#[derive(Accounts)]` in `programs/shingo/src/lib.rs`.

No account context uses Anchor's `close = recipient` directive. As a result, `Season`, `SeasonEscrow`, `Signal`, `RevealedSignal`, and `SubscriptionPass` PDAs all persist forever after their useful life ends. The rent reserve allocated at init for each account is never reclaimed.

For a busy trader (50 seasons over time, 100 subscribers per season, 30 signals per season), this is approximately:

- 50 Season PDAs
- 50 SeasonEscrow PDAs (after C-03 fix, can be closed at season close)
- 50 × 30 = 1,500 Signal PDAs
- 50 × 30 = 1,500 RevealedSignal PDAs
- 50 × 100 = 5,000 SubscriptionPass PDAs

Each PDA has a rent-exempt reserve of ~0.001 to 0.002 SOL depending on data size, totalling several SOL per active trader over a year of use. Not a security primary, but worth flagging for operational sustainability.

#### Recommended fix

- Add `close = trader` to `SeasonEscrow` in `CloseSeason` (this is also part of the C-03 fix).
- Optionally add `close = trader` to `Season` in `CloseSeason` if historical Season state is not needed.
- Consider a `close_signal(signal_id)` instruction the trader can call after reveal to reclaim Signal and RevealedSignal rent.
- Consider a `claim_back_pass()` instruction subscribers can call to reclaim their pass rent.

### M-08 `computation_offset` is user-supplied and unbound, concurrent callers can DoS each other's MPC computation

**Severity**: Medium.
**Files**: `programs/shingo/src/lib.rs:1113-1120, 1248-1253, 590-595, 736-742`.

The `decrypt_signal` and `reveal_signal` instructions take `computation_offset: u64` as a user-supplied parameter. The `computation_account` PDA in both account contexts is derived via `derive_comp_pda!(computation_offset, mxe_account, ErrorCode::ClusterNotSet)`. Per `arcium-client 0.9.7`, the comp PDA derivation namespaces only on `(cluster_offset, computation_offset)` and not on the caller's pubkey. So two different shingo callers using the same offset against the same Arcium cluster collide on the same PDA.

```rust
#[account(
    mut,
    address = derive_comp_pda!(computation_offset, mxe_account, ErrorCode::ClusterNotSet)
)]
pub computation_account: UncheckedAccount<'info>,
```

#### Impact

Attacker watches the RPC mempool, sees Alice's pending `decrypt_signal` with `computation_offset = 12345`, then submits her own `decrypt_signal` with the same offset. The Arcium runtime initializes the computation account at the colliding PDA for the attacker. Alice's tx then fails to re-init the same account and reverts.

Cost to attacker is the rent reservation for the comp account (reclaimable via Arcium's `claim_computation_rent`). Cost to Alice is a failed tx and the need to retry with a different offset. Repeated, this is a DoS on subscriber decryption.

#### Recommended fix

Either bind the computation_offset to the caller, or maintain an on-chain monotonic counter inside `TraderAccount` or `SubscriptionPass` and require `computation_offset == counter + 1` with an automatic increment. Simplest sketch:

```rust
let expected_offset = ctx.accounts.follower_pass.next_offset
    .checked_add(1)
    .ok_or(ShingoProgramError::CheckedArithmeticFailure)?;
require!(computation_offset == expected_offset, ShingoProgramError::InvalidComputationOffset);
ctx.accounts.follower_pass.next_offset = expected_offset;
```

This requires adding `next_offset: u64` to `SubscriptionPass`.

The `.unwrap_or(current_season)` pattern reads as if it were a soft fallback, when in fact it is dead code. Cleaner:

```rust
seeds = [SeasonEscrow::SEED, trader.key().as_ref(),
    (trader_account.current_season + 1).to_le_bytes().as_ref()]    // will panic on overflow under overflow-checks=true; acceptable
```

Or accept that this code is unreachable and document.

---

### I-09 `reveal_signal` is permissionless, document the design intent

**Severity**: Informational.
**File**: `programs/shingo/src/lib.rs:691-769`.

Any signer paying the Arcium fee can trigger `reveal_signal` for any closed-season signal. The architecture's "after-close = public" position makes this intentional, but combined with C-07 (cross-trader reveal) the same permissionless trigger lets an attacker leak any private signal once a same-id closed season exists.

Document the intent. If reveals are intended to be permissionless, explicitly note that closing a season exposes every signal of that season to everyone. If trader-gated reveal is intended, add a `require!(signer == season.trader, ...)`.

## Compound attack chains

Several findings compose into worse failure modes. Each chain lists its preconditions (other findings whose fixes must be applied first for the chain to become reachable), the attack flow, and the finding whose fix disarms the chain.

### Chain X-1 (Critical): full rug after escrow fix

Preconditions: C-01, C-02, C-03 fixed.

1. Trader Alice runs `initialize_season(price=1 SOL, spots=1000, minimum_count=0)`. H-06 unfixed lets her set 0.
2. Subscribers pay into the escrow. With C-05 fixed, this works.
3. Alice calls `close_season`. C-04's inverted check accepts because `count (0) < minimum (0)` is false in the natural fix, but if C-04 is also fixed in the natural direction (`count >= minimum`), the `0 >= 0` test passes anyway. Close succeeds.
4. Alice walks away with the escrow. Subscribers got zero signals.

Mitigation: fix H-06 to require `minimum_count >= 1` (or higher), so the rug requires actually publishing at least one signal.

### Chain X-2 (Critical): one subscription decrypts everything

Preconditions: C-05 fixed.

1. Mallory subscribes to any one cheap season (cost as low as 100 lamports + tip).
2. For every premium signal in the protocol, Mallory calls `decrypt_signal` passing her pass for the cheap season and the target signal account.
3. The MPC decrypts every signal, re-encrypted for Mallory's receiver pubkey.

The protocol's per-trader-per-season pricing model is bypassed by a single cheap subscription.

Mitigation: fix C-08 (bind `signal`, `season`, `trader_account` to the same trader via seeds and `signal.metadata.author` check).

### Chain X-3 (Critical): cross-trader signal reveal

Preconditions: C-01, C-02, C-03 fixed.

1. Mallory initializes her own trader and runs `initialize_season(price=1, spots=1, minimum_count=1)`.
2. Mallory publishes one signal of her own to bring `count == minimum`.
3. Mallory calls `close_season`. Succeeds with the fixed close path.
4. Mallory's season has `id == 1` (her first season), `is_active == false`.
5. Victim Alice is running her own active season 1 with secret signals.
6. Mallory calls `reveal_signal` passing her closed season 1 + Alice's secret signal. The require accepts because `1 == 1` and `!is_active`. Arcium decrypts and emits the plaintext.

Mitigation: fix C-07 (bind `season` to `signal.metadata.author` via PDA derivation).

### Chain X-4 (High): post-close manipulation

Preconditions: H-04 unfixed.

1. Trader Alice closes season 3 normally.
2. The market moves. Alice now knows what signal would have been profitable.
3. Alice (or anyone, via C-06) calls `encrypt_signal` against season 3 (which is `is_active=false`). The function accepts because there is no is_active check.
4. Alice now has a "historical" signal she can `reveal_signal` to demonstrate her supposed prescience.

Mitigation: fix H-04 (add is_active check) and C-06 (bind season to signer).

### Chain X-5 (Critical): trader-side fund-bricking

Preconditions: today, with no fixes needed except making the protocol actually usable.

1. Alice runs season 3, accepts subscriber funds.
2. Alice calls `initialize_trader_account` again. State resets to `current_season=0, has_active_season=false`.
3. Alice's season 3 escrow is now unreachable. No path forward closes it. Subscribers' funds are stuck.

Mitigation: fix C-09 (`init` not `init_if_needed`).

### Chain X-6 (High): count-bump lockout

Preconditions: C-06 live (encrypt_signal.season unconstrained), C-04 inverted (still live in HEAD).

1. Victim Alice runs a season with `minimum_number_of_episodes = 5`.
2. Alice publishes 4 signals. `season.count = 4`.
3. Mallory calls `encrypt_signal` with Alice's season and Mallory as signer. Mallory pays for a useless Signal at her own PDA. Alice's `season.count = 5`.
4. With the inverted minimum check (`count < minimum`), Alice's close now fails (`5 < 5` is false).
5. Alice cannot close. There is no decrement primitive. Alice's escrow funds are stuck.

If C-04 is fixed instead (`count >= minimum`), the same count-bump enables Alice to close earlier than she intended, but only if Alice herself wants to close, so the threat profile changes from DoS to "loss of control over close timing". Either way, the unconstrained encrypt_signal.season + count being a critical gate is a problem.

Mitigation: fix C-06 (bind `season` to signer) and C-04 (correct the direction of the check).

### Chain X-7 (Medium): computation_offset front-run DoS

Preconditions: M-08 live (offset unbound, current state).

1. Attacker watches the RPC mempool for shingo decrypt_signal / reveal_signal transactions.
2. When a victim's tx with `computation_offset = X` is pending, the attacker submits their own decrypt with the same offset.
3. The Arcium runtime initializes the computation account at the colliding PDA for the attacker. Victim's tx reverts.
4. Attacker can reclaim their rent via Arcium's `claim_computation_rent`. Cost to attacker is near-zero, cost to victim is a failed tx and retry.

Mitigation: fix M-08 (bind offset to caller-derived counter or signer pubkey).

### Chain X-8 (Medium): pre-reveal PDA squat

Preconditions: C-01 to C-03 fixed (close_season works), C-07 live (cross-trader reveal).

1. Victim Alice publishes a signal during her active season N.
2. Mallory creates and closes her own season N.
3. Mallory calls `reveal_signal` with her closed season and Alice's signal. The `RevealedSignal` PDA is at `[revealed, Alice, N, k]` (keyed at Alice's address, paid by Mallory because of `init_if_needed`).
4. Mallory's reveal succeeds. Alice's plaintext is now public, before Alice ever closed her own season.

Mitigation: fix C-07 (bind `season` to `signal.metadata.author` in PDA derivation).

### Chain X-9 (High): close_season cross-account state corruption

Preconditions: H-03 live (close_season.trader_account unconstrained), C-01 to C-03 fixed.

1. Mallory runs a few throwaway seasons of her own to arrive at her current_season = N matching some victim Alice's current_season.
2. Mallory calls `close_season` with `trader = Mallory` (signer), `trader_account = Alice's TA` (unconstrained), `season = [Season, Mallory, N]` (her own active season), `season_escrow = [escrow, Mallory, N]`.
3. The guards pass: Mallory's season is hers (post C-01 fix), Alice's TA has `has_active_season = true`. The function then writes `Alice.trader_account.has_active_season = false; Alice.trader_account.signal_count = 0`.
4. Mallory's own escrow is drained to her wallet. Alice's TA is corrupted: she cannot close her own season because `has_active_season = false`, and `close_season` requires it.
5. Alice's escrow funds are permanently locked. Mallory paid the cost of running and closing one throwaway season.

Mitigation: fix H-03 (constrain `trader_account` to `[TraderAccount::SEED, trader.key()]`).
